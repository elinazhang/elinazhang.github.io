About MVVM, MVP and MVC

Reading Notes

MVC (Model-View-Controller)

MVC is one of the most common used structure. It defines three components:

  • View - the UI
  • Controller - contains the logic that alters the model
  • Model - for storing data

And communication between the three parts works as indicated above.

  • View passes calls to Controller.
  • Controller manipulates Model to change.
  • Model sends the updated data back to the View, where users can get response.

There are other commonly used working patterns, eg.

  • giving order to Controller directly -> manipulates the Model -> updates the View

But they share one thing in common, communication goes one-way, it always follows:

  • View -> Controller -> Model -> View.

Controller will not contact View directly as it doesn’t know anything about View, and View, on the other hand, can call different Controllers.

MVP (Model-View-Presenter)

MVP replaces Controller with Presenter, by doing so, it changes the communication direction. Presenter can refer back to the View while Controller can’t.

In MVP, the relationship between Controller & View is described as Supervising Controller & Passive View.

View becomes really thin as it only contains basic logic or no logic at all. Controller handles all the complex logic.

MVVM (Model–View-ViewModel)

MVVM replaces Presenter with ViewModel. It uses data-binding. Any change on Model will be represented in ViewModel, and any change in ViewModel, will be reflected on View.
